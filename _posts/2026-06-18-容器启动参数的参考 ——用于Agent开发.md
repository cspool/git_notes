# 容器启动参数的参考 ——用于Agent开发

## 主要贡献

出发点：host 上用普通用户登录，或者在多人服务器上只有普通用户权限，但希望进入容器后能用 root 权限高效做 agent 开发，同时又不要把 host 系统、host HOME、host 包环境暴露给容器随意修改。

“安全隔离”和“高权限开发”同时成立：

- 容器里需要 root，方便 agent 安装依赖、改环境、生成文件、跑调试命令。
- host 侧仍然需要安全，不能因为容器 root 误操作就覆盖 host 包、host 软件或共享 skill。
- 容器 root 生成的项目文件，回到 host 后普通用户还要能继续编辑、删除和维护。
- Claude/Codex 这类 agent 工具需要保留登录态和配置写入能力，否则开发体验会断。

脚本的核心设计是把写权限拆开：

- 项目目录和容器缓存保持 `rw`，保证训练、调试、生成文件都能正常落盘。
- host 包、host 软件、共享 skill 默认 `ro`，容器可以复用，但不能覆盖 host 侧环境。
- 容器默认 root 登录，同时用 ACL 让 host 用户继承共享 `rw` 目录的读写权限。
- Claude/Codex 的个人配置保持 `rw`，保证工具能登录、更新状态；但外部共享 skill 和插件缓存仍按只读保护。

最终效果是：agent 在容器里像在一台有 root 权限的开发机上工作，但风险被限制在容器和明确允许写入的目录内；host 普通用户不需要把整个 HOME 暴露出去，也不会因为容器 root 生成文件而失去项目目录控制权。

## 注意点速览

- `PROJECT_DIR` 是项目工作区，默认 `rw`，容器内代码修改和实验输出都会写回 host 项目目录。
- `CONTAINER_HOME_CACHE_DIR`、`CACHE_DIR`、`NPM_GLOBAL_DIR` 是容器专用可写目录，不等同于 host 真实 HOME。
- host 包目录、host 软件目录、NVIDIA 工具、skill symlink 目标默认 `ro` 或 `ro,rshared`，用于复用而不是修改。
- host HOME 下的只读共享目录采用旁路挂载，不覆盖容器 `$HOME` 原路径；需要命令发现时再把旁路目录加入 `PATH`。
- 不挂载整个 host HOME，只选择性挂载 Claude/Codex 配置。
- `.claude`、`.claude.json`、`.codex` 默认 `rw`，因为 agent 工具需要更新登录态、配置和本地状态。
- `.bashrc` 和外部共享 skill/插件缓存保持只读，避免容器影响 host shell 或共享资源。
- 容器默认 root 登录，但会给可写共享目录设置 ACL，让 host 普通用户继承读写权限。
- `--privileged` 不默认启用，除非短时隔离调试且没有更窄权限方案，否则应避免使用。

这份说明只讨论 `docker/run_other_project_gpu.sh` 里最关键的权限设计：
哪些目录应该共享、哪些目录应该只读、为什么容器默认 root 登录时 host 用户仍然能继续修改共享目录，以及为什么 Claude/Codex 配置需要读写挂载。

## 1. `-v` 挂载对象：项目可写，共享资源只读

容器里最重要的一条原则是：项目目录给当前任务读写，host 上已有的软件、包、skill 只给只读。

项目目录是主要工作区，因此直接读写挂载：

```text
-v ${PROJECT_DIR}:/workspace/${PROJECT_NAME}
```

容器中的代码修改、实验输出、日志、项目内虚拟环境等都会落在这个目录里。它是当前项目自己的工作空间，所以默认 `rw`。

容器自己的 HOME、缓存目录和 npm 全局目录也是可写的，但它们不是 host 的真实 HOME，而是脚本单独准备的缓存目录：

```text
-v ${CONTAINER_HOME_CACHE_DIR}:${CONTAINER_HOME}
-v ${CACHE_DIR}:${CONTAINER_CACHE_DIR}
-v ${NPM_GLOBAL_DIR}:${CONTAINER_NPM_GLOBAL_DIR}
```

这样容器可以写 Hugging Face、Torch、Triton、npm 等缓存，但不会把整个 host HOME 暴露给容器。

相反，host 上已有的软件、包目录、NVIDIA 工具和 skill 目标默认只读：

```text
-v ${HOST_NVIDIA_DIR}:${CONTAINER_NVIDIA_DIR}:ro,rshared
```

脚本还会把常见 host 包目录按只读方式挂进容器，例如：

```text
${HOST_HOME}/.cache
${HOST_HOME}/.local
${HOST_HOME}/.conda, .mamba, .micromamba
${HOST_HOME}/.npm, .nvm
${HOST_HOME}/.cargo, .rustup
${HOST_HOME}/.venv, .virtualenvs, venv
${HOST_HOME}/anaconda3, miniconda3
node_modules, site-packages, dist-packages, __pypackages__
```

默认模式是：

```text
BLOCKED_PATH_MODE=ro,rshared
```

这类目录通常属于 host 侧环境或工具缓存。容器可以读取它们来复用已有环境，但不能覆盖、升级、删除 host 上的副本。这样可以避免容器里的 root 或 agent 误操作，把 host 普通用户依赖的包或工具链改坏。

对 host HOME 下的只读 bind mount，推荐采用旁路挂载，而不是直接覆盖容器 `$HOME` 下的同名路径：

```text
CONTAINER_HOST_HOME_DIR=${CONTAINER_HOME}/.host-home
HOST_HOME_READONLY=${CONTAINER_HOST_HOME_DIR}

${HOST_HOME}/.local    -> ${HOST_HOME_READONLY}/.local:ro,rshared
${HOST_HOME}/.conda    -> ${HOST_HOME_READONLY}/.conda:ro,rshared
${HOST_HOME}/miniconda3 -> ${HOST_HOME_READONLY}/miniconda3:ro,rshared
${HOST_HOME}/.bashrc   -> ${HOST_HOME_READONLY}/.bashrc:ro
```

这样容器自己的 `${CONTAINER_HOME}/.local`、`${CONTAINER_HOME}/.cache`、`${CONTAINER_HOME}/.conda` 仍然来自 `CONTAINER_HOME_CACHE_DIR`，可以被容器写入；host 侧副本仍可通过 `${HOST_HOME_READONLY}/...` 读取。对于需要复用 host 用户安装命令的场景，只把可执行目录作为 PATH 后备加入：

```text
PATH=${CONTAINER_HOME}/.local/bin:${HOST_HOME_READONLY}/.local/bin:${CONTAINER_NPM_GLOBAL_DIR}/bin:...
```

这比把 `${HOST_HOME}/.local` 直接挂到 `${CONTAINER_HOME}/.local` 更稳：容器内 `pip install --user`、用户级工具缓存和 shell 初始化文件不会被 host 只读目录挡住；host `.bashrc` 或 host `miniconda3` 里的 conda init 也不会覆盖容器镜像自己的 conda 环境。

skill 的处理也遵循同样原则。Claude/Codex 配置目录里如果有 skill symlink，例如：

```text
~/.claude/skills/foo -> ~/.orchestra/skills/foo
```

脚本会解析 symlink 目标，并默认把目标 skill 根目录只读挂载进容器：

```text
SKILL_SYMLINK_TARGET_MODE=ro
```

也就是说，容器可以使用 host 上已有的 skill，但默认不能修改这些 skill。对于共享服务器或多个 agent 会话复用同一套 skill 的场景，这一点很重要。

## 2. root 登录容器：host 用户继承共享 rw 目录权限

脚本默认让容器以 root 身份登录：

```text
CONTAINER_LOGIN_USER=root
--user 0:${HOST_GID}
```

这样容器里安装包、调试环境、处理系统权限会更方便。同时，脚本会把 host 用户的附加组也加进容器：

```text
--group-add <host supplementary groups>
```

问题是：容器 root 在共享目录里创建的文件，host 普通用户是否还能继续编辑？

脚本用 ACL 解决这个问题。在容器启动前，它会对可写共享目录设置默认 ACL：

```text
u:${HOST_USER}:rwX
d:u:${HOST_USER}:rwX
```

覆盖的重点目录包括：

```text
${PROJECT_DIR}
${CACHE_DIR}
${CONTAINER_HOME_CACHE_DIR}
${NPM_GLOBAL_DIR}
```

效果是：即使文件是在容器内由 root 创建的，host 上的普通用户也会继承读写权限。文件显示的 owner 可能仍是 `root:${HOST_GROUP}`，或者在某些环境下显示为 remapped owner，但 host 用户仍然可以通过 ACL 读写。

容器启动 shell 还会设置：

```text
umask 0002
```

不过后续手动 `docker exec ... bash` 进入的新 shell 不一定继承同样的 umask。因此，真正保证 host 用户可写的是目录上的默认 ACL，而不是单纯依赖 umask。

如果不想用 root 登录，可以切回 host 用户模式：

```bash
CONTAINER_LOGIN_USER=host RECREATE=1 bash docker/run_other_project_gpu.sh
```

## 3. Claude/Codex 配置共享：配置目录保持 rw

脚本不会把整个 host HOME 挂进容器，而是只选择性挂载 Claude/Codex 相关配置：

```text
${HOST_HOME}/.claude -> ${CONTAINER_HOME}/.claude:${CLAUDE_CONFIG_MODE:-rw}
${HOST_HOME}/.claude.json -> ${CONTAINER_HOME}/.claude.json:${CLAUDE_CONFIG_MODE:-rw}
${HOST_HOME}/.codex -> ${CONTAINER_HOME}/.codex:${CODEX_CONFIG_MODE:-rw}
${HOST_HOME}/.bashrc -> ${HOST_HOME_READONLY}/.bashrc:ro
```

Claude 和 Codex 的配置目录默认 `rw`，原因是这些工具会正常更新自己的状态，例如：

- 登录态和认证信息
- `config.toml` 等配置文件
- 工具自身的本地状态
- 用户安装或更新的个人级配置

如果把 `.claude` 或 `.codex` 做成只读，工具在容器里可能无法登录、无法更新配置，甚至因为写临时状态失败而异常退出。

这里的边界是：

- Claude/Codex 配置目录：默认 `rw`，让工具能正常工作。
- `.bashrc`：默认 `ro` 且走 `${HOST_HOME_READONLY}` 旁路，避免覆盖容器 shell 初始化逻辑，也避免容器随意改 host shell 初始化文件。
- Claude/Codex 配置目录中指向外部共享 skill 或插件缓存的路径：默认按只读保护，避免覆盖共享资源。

总结起来，这个启动脚本的权限模型是：

```text
当前项目和容器缓存：rw
host 包、host 软件、共享 skill：ro
Claude/Codex 个人配置：rw
host HOME 整体：不挂载
root 容器创建的共享文件：通过 ACL 让 host 用户继续可写
```
